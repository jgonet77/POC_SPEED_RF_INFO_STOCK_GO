🤖Génération multi-agents Opus 4.8

Plutôt que de générer le code séquentiellement, un workflow multi-agents a été lancé : 9 agents Opus 4.8 ont travaillé en parallèle, chacun responsable d'une couche précise. Un contrat partagé (types, noms de fonctions, chemins d'import) a été injecté dans chaque prompt pour garantir la cohérence sans dépendance entre agents.

⚡ Phase 1 — Foundation (3 agents en parallèle) Fondations
go.mod + config/config.go models/ (4 fichiers) services/token_store.go
🗄️ Phase 2 — Data Layer (2 agents en parallèle) Accès données
db/db.go + 4 repositories services/health_service.go + stock_service.go
🌐 Phase 3 — API Layer (2 agents en parallèle) API REST
middleware/auth.go routes/auth.go + stock.go + activity.go + health.go
🔧 Phase 4 — Assembly (1 agent) Point d'entrée
main.go (Fiber app, CORS, graceful shutdown)
🔍 Phase 5 — Review (1 agent) Validation
Vérification imports, types, SQL @p1/@p2, JSON tags, null safety… 0 incohérence détectée
💡
Pourquoi des agents parallèles ? Chaque couche de l'architecture est indépendante des autres à condition de partager un contrat commun (noms de types, signatures de fonctions, chemins d'import). En parallélisant, le temps de génération passe de ~60 min séquentiel à ~15 min. Le review agent final joue le rôle d'intégration continue : il lit tous les fichiers et corrige les éventuelles divergences.

🗄️Problème de connexion SQL Server — TCP vs Named Pipes

⚠️
Symptôme rencontré Le serveur Go démarrait mais échouait à se connecter à SQL Server avec l'erreur : unable to open tcp connection with host 'LAPVMI116:1433'. Le backend Python se connectait sans problème à la même base.

Pourquoi Python fonctionnait et Go non ?

🐍 Python — pyodbc + ODBC Driver 17

pyodbc utilise le pilote ODBC de Microsoft qui supporte deux protocoles :

Named Pipes (par défaut local) : connexion via le système de fichiers Windows (\\.\pipe\MSSQL$SQL2019\sql\query). Ne nécessite pas que TCP soit activé.

TCP/IP : connexion réseau standard sur port 1433. Nécessite que TCP soit activé dans SQL Server.

Quand le serveur est local, ODBC préfère automatiquement les Named Pipes — d'où le fait que Python fonctionnait sans configuration TCP.

🐹 Go — go-mssqldb (pur Go)

go-mssqldb est un driver 100% Go, sans dépendance au pilote ODBC de Windows. Il utilise uniquement TCP/IP pour se connecter à SQL Server.

Pour une instance nommée comme LAPVMI116\SQL2019, le driver interroge le SQL Server Browser (UDP 1434) pour découvrir le port dynamique assigné à l'instance, puis se connecte en TCP.

Si TCP n'est pas activé dans SQL Server, la connexion échoue immédiatement — même si le serveur est sur la même machine.

Solution appliquée : activer TCP/IP dans SQL Server Configuration Manager

SQL Server Configuration Manager → Protocols for SQL2019 Double-clic sur TCP/IP → onglet Protocol → Enabled = Yes
Configurer le port fixe Onglet IP Addresses → IPAll → TCP Dynamic Ports = vide, TCP Port = 1433
Redémarrer le service SQL Server (SQL2019) SQL Server Services → SQL Server (SQL2019) → clic droit → Restart
Activer SQL Server Browser SQL Server Services → SQL Server Browser → Start (permet la découverte de l'instance nommée)

Correction dans le code Go — format du DSN

Une seconde erreur s'est produite lors du premier essai : l'URL générée encodait le / en %2F, ce qui rendait le DSN invalide. Pour les instances nommées, go-mssqldb attend le paramètre instance= en query string, pas un slash dans le host.

❌ DSN incorrect (encodage URL)

sqlserver://sa:BKS@LAPVMI116%2FSQL2019
  ?database=POC_V8SpeedDeveloppement

✅ DSN correct (query param)

sqlserver://sa:BKS@LAPVMI116
  ?database=POC_V8SpeedDeveloppement
  &instance=SQL2019

Risques d'activer TCP sur SQL Server

RisqueNiveauMitigation
Exposition du port 1433 sur le réseau local Modéré Restreindre via le pare-feu Windows aux seules IP autorisées (serveur API uniquement)
Tentatives de connexion brute-force depuis le réseau Modéré Utiliser un compte SQL dédié (pas sa) avec mot de passe fort en production
Exposition à Internet Élevé si mal configuré Ne jamais ouvrir le port 1433 vers Internet. L'API Go sert d'intermédiaire — seul le port 8000 doit être accessible depuis le téléphone.
Impact sur les connexions Python existantes Aucun Activer TCP ne désactive pas les Named Pipes. Les deux coexistent.
🛡️
Bonne pratique adoptée Une règle de pare-feu "Allow FastAPI 8000" existait déjà. Le port 1433 reste fermé vers l'extérieur. Seul le port 8000 (API Go) est accessible depuis les téléphones sur le réseau local. SQL Server n'est jamais exposé directement.

📱Connexion Android — HTTP en clair sur réseau local

⚠️
Symptôme rencontré L'app Android affichait : "api_test_failure: cleartext communication to 192.168.1.20 not permitted by network security policy". L'app fonctionnait au bureau mais pas à la maison.

Pourquoi ça fonctionnait au bureau et pas à la maison ?

Depuis Android 9 (API 28), toutes les communications HTTP en clair sont bloquées par défaut. Il est possible d'autoriser des exceptions via un fichier network_security_config.xml.

EnvironnementAdresse utiliséeRésultat avec l'ancienne config
Bureau (réseau entreprise) Nom DNS — ex. LAPVMI116 Fonctionnait — la balise <domain> accepte les noms DNS
Maison (réseau WiFi privé) Adresse IP — 192.168.1.20 Bloqué — Android ignore les IPs dans <domain>
🚫
Limitation Android documentée La balise <domain includeSubdomains="true">192.168.1.20</domain> est silencieusement ignorée par Android. Seuls les noms de domaine (DNS) sont valides dans cette balise — jamais des adresses IP.

Solution : Build Variants Android (Debug vs Release)

La solution propre est d'utiliser le système de variants de build d'Android. Chaque variant peut avoir ses propres ressources. Le build Debug (utilisé pendant les tests) prend automatiquement le fichier dans src/debug/ en priorité sur src/main/.

📁 src/debug/res/xml/network_security_config.xml
(builds de développement) 

<!-- HTTP autorisé partout en debug -->
<network-security-config>
  <base-config
    cleartextTrafficPermitted="true">
    <trust-anchors>
      <certificates src="system" />
    </trust-anchors>
  </base-config>
</network-security-config>

📁 src/main/res/xml/network_security_config.xml
(builds de production) 

<!-- HTTPS obligatoire en production -->
<network-security-config>
  <domain-config
    cleartextTrafficPermitted="true">
    <domain>localhost</domain>
    <domain>10.0.2.2</domain>
  </domain-config>
  <base-config
    cleartextTrafficPermitted="false" />
</network-security-config>
✅
Résultat Le build Debug (installé via Android Studio ou gradlew installDebug) autorise HTTP en clair vers n'importe quelle adresse IP ou domaine — chez soi, au bureau, sur n'importe quel réseau. Aucune modification de code n'est nécessaire en changeant de réseau. Il suffit de mettre à jour l'IP du serveur dans les Paramètres de l'application.
