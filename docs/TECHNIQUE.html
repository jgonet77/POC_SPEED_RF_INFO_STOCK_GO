Architecture et Implémentation - Expliquée simplement

🎯Avant de lire ce document

Cette doc explique comment fonctionne l'app techniquement (en détail). Les principes sont expliqués avec des analogies simples, comme si tu enseignais ça à un copain de classe!

🏗️1. Architecture générale (Le plan du bâtiment)

Imagine une maison avec 3 étages :

Étage 1: Le Salon (FRONTEND)

C'est l'app Android sur ton téléphone. C'est ce que tu VOIS et ce que tu TOUCHES.

Techniquement : Kotlin (langage de programmation Android)

Analogie : C'est la façade d'un restaurant (ce que les clients voient)

Étage 2: La Cuisine (BACKEND)

C'est le serveur Python qui fait le travail. Tu ne le vois pas, mais c'est LUI qui travaille.

Techniquement : Python (FastAPI) - API REST

Analogie : C'est la cuisine d'un restaurant (où on prépare à manger)

Étage 3 (Sous-sol): L'Entrepôt (DATABASE)

C'est la base de données SQL Server du WMS SPEED. C'est où on stocke TOUTES les infos (stock, utilisateurs, emplacements, etc.)

Techniquement : Microsoft SQL Server

Analogie : C'est le vrai entrepôt (où on stocke les vrais produits)

TU (sur ton téléphone Android) ↓ (Tu cliques sur "Chercher") [FRONTEND - App Kotlin] ↓ (Envoie une requête HTTP) [BACKEND - Serveur Python] ↓ (Fait une requête SQL) [DATABASE - SQL Server] ↓ (Retourne les données) [BACKEND - Serveur Python] ↓ (Formate les données) [FRONTEND - App Kotlin] ↓ (Affiche les résultats) TU VOIS LES RÉSULTATS! 🎉

🔧2. Les 3 couches du serveur (Architecture en couches)

Le serveur Python (backend) est divisé en 3 étages. Chaque étage a un travail spécifique:

Couche 1: API LAYER (Réceptionniste 📞)

Rôle : Recevoir les requêtes du téléphone et envoyer les réponses.

Analogie : Un réceptionniste d'hôtel qui reçoit tes appels téléphoniques.

@app.post("/api/auth/login") def login(credentials: LoginRequest): # Reçoit login/password du téléphone return {"status": "success", "token": "xyz123"}

Questions qu'elle se pose : "D'où vient cette requête? Comment la traiter?"

Couche 2: SERVICE LAYER (Le Chef 👨‍🍳)

Rôle : Faire la logique métier (les règles du jeu). "Ce qu'on DOIT faire" pour un client.

Analogie : Un chef qui décide COMMENT préparer un plat.

class AuthService: def authenticate_user(login: str, password: str): # Vérifier le mot de passe # Récupérer les activités de l'utilisateur return user_data

Questions qu'il se pose : "Est-ce que ce login/password est correct? Qui est cet utilisateur?"

Couche 3: REPOSITORY LAYER (L'Entrepôt 📦)

Rôle : Communiquer UNIQUEMENT avec la base de données. "Aller chercher ou sauver des données"

Analogie : L'entrepôtier qui va chercher les ingrédients dans l'entrepôt.

class UserRepository: def find_by_login(login: str): # SELECT * FROM Utilisateurs WHERE login = ? return database_result

Questions qu'il se pose : "Comment accéder à la base de données? Quels sont les bonnes requêtes SQL?"

Pourquoi 3 couches et pas 1 seule?

Avantage 1: Si la base de données change, on n'a qu'à changer la Repository, pas tout le reste!
Avantage 2: On peut tester chaque couche indépendamment (tests unitaires).
Avantage 3: C'est plus facile à comprendre et à maintenir.

🛣️3. Les endpoints API (Les "chemins" de communication)

Un endpoint API, c'est comme une route postale. Tu envoies un "colis" (requête HTTP) à une adresse (endpoint), et le serveur te répond.

Endpoint 1: Se connecter (Login)

POST /api/auth/login

Qui demande : L'app Android

Qu'on envoie (REQUEST) :

{ "login": "jean.dupont", "password": "secret123" }

Qu'on reçoit (RESPONSE) :

{ "status": "success", "user_id": 42, "activities": [ {"id": 1, "name": "Préparation Commande"}, {"id": 2, "name": "Inventaire"} ] }

Endpoint 2: Chercher un SKU

GET /api/stock/search?sku=CHAISE-BLEUE

Qui demande : L'app Android

Paramètres : Le SKU du produit qu'on cherche

Qu'on reçoit (RESPONSE) :

{ "results": [ { "sku": "CHAISE-BLEUE", "name": "Chaise Bleue Premium", "quantity": 45 } ] }

Endpoint 3: Voir les détails d'un produit

GET /api/stock/CHAISE-BLEUE/details

Qui demande : L'app Android

Qu'on reçoit (RESPONSE) :

{ "sku": "CHAISE-BLEUE", "name": "Chaise Bleue Premium", "quantity": 45, "location": { "zone": "A", "rack": 5, "position": 3 }, "additional_info": { "lot": "2024-001", "expiration_date": "2025-12-31", "status": "BON" } }

📱4. Le Frontend (App Android Kotlin)

L'app Android est ce que tu VOIS. Elle a plusieurs parties:

Activities (Les écrans)

Chaque écran est une "Activity"

Exemples :

  • LoginActivity
  • HomeActivity
  • SearchActivity
  • DetailActivity

ViewModels (Le cerveau)

C'est qui gère la logique de chaque écran

Exemples :

  • LoginViewModel
  • SearchViewModel
  • DetailViewModel

API Client (Retrofit)

C'est qui communique avec le serveur Python

Le langage : HTTP REST (comme un SMS entre ton téléphone et le serveur)

Repository (Cache local)

Stockage local sur le téléphone (SharedPreferences)

Utile pour : Sauver le token de connexion

Utilisateur clique sur "Chercher" ↓ [Activity] demande au [ViewModel] ↓ [ViewModel] appelle le [API Client (Retrofit)] ↓ [API Client] envoie une requête HTTP au serveur Python ↓ [Serveur Python] traite et répond avec les données ↓ [ViewModel] reçoit les données et les envoie à [Activity] ↓ [Activity] affiche les résultats à l'écran ↓ L'utilisateur VOIT les résultats! 🎉

💾5. La Base de Données (SQL Server)

C'est là qu'on stocke TOUTES les infos du WMS SPEED. Les tables principales sont :

Table Contient Exemple
Utilisateurs Login, password, infos employé jean.dupont, password123
Articles/SKU Codes produits, noms, descriptions CHAISE-BLEUE, Chaise Bleue Premium
Stock Quantités par SKU 45 Chaises Bleues disponibles
Emplacements Où les produits sont rangés (Zone, Rack, Position) Zone A, Rack 5, Position 3
Lots Numéro de lot, date d'expiration Lot 2024-001, Expire 2025-12-31
Activités Préparation, Inventaire, etc. Préparation Commande

📋 Note:

La documentation complète des tables SQL Server sera fournie à l'étape d'analyse technique. Pour le POC, on crée une structure de test simplifiée.

🔐6. Comment la sécurité fonctionne

Utilisateur se connecte ↓ [App Android] envoie login + password au serveur (HTTPS - connexion sécurisée) ↓ [Serveur Python] vérifie les identifiants dans la BD ↓ SI correct → Serveur crée un TOKEN (clé unique) SI incorrect → Serveur dit "Erreur 401: Non autorisé" ↓ [App Android] reçoit le TOKEN et le sauvegarde ↓ Pour les futures requêtes, [App Android] envoie le TOKEN dans chaque demande ↓ [Serveur Python] vérifie le TOKEN avant de répondre ↓ Si TOKEN valide → Traiter la requête Si TOKEN invalide → Dire "Erreur 401: Non autorisé"

⚠️ Important:

On JAMAIS envoyer le mot de passe pour chaque requête. On utilise un TOKEN (jeton) une seule fois.
C'est comme montrer une carte d'identité une fois à la banque, au lieu de dire ton mot de passe à chaque retrait!

🛠️7. Les outils et technologies

Backend

  • Python 3.10+ - Langage
  • FastAPI - Framework pour l'API
  • pyodbc - Connexion à SQL Server
  • Pydantic - Validation des données
  • pytest - Tests unitaires

Frontend

  • Kotlin - Langage Android
  • Android Studio - IDE (éditeur de code)
  • Retrofit - HTTP Client
  • ViewModel & LiveData - Gestion de l'état UI
  • Espresso - Tests d'interface

📝8. Résumé en images

L'architecture en 1 phrase

"L'app Android sur ton téléphone parle à un serveur Python via HTTP. Le serveur parle à la base de données SQL Server pour récupérer les infos du stock. Puis il envoie les résultats à l'app, qui les affiche à l'écran."

Composant Rôle Technologie
Frontend Ce que tu VOIS et TOUCHES Kotlin (Android natif)
Backend (Couche API) Reçoit les requêtes du téléphone Python FastAPI
Backend (Couche Service) Fait la logique métier Python (ServiceClasses)
Backend (Couche Repository) Parle à la base de données Python (pyodbc)
Database Stocke toutes les infos Microsoft SQL Server
